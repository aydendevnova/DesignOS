# Cursor Validation Handoff

This document defines the handoff protocol from Claude Code (generation) to Cursor IDE (visual validation). Claude Code generates the code and runs the anti-slop checklist. Cursor receives the output with a validation report and uses its browser tools to verify visually.

## When This Handoff Occurs

After Claude Code completes code generation and runs the anti-slop enforcement checklist, it produces:
1. The generated Next.js project (code files).
2. A `VALIDATION-REPORT.md` file in the project root with flagged issues and the checklist results.

Cursor picks up from here.

## VALIDATION-REPORT.md Format

Claude Code generates this file at the project root after every generation pass.

```markdown
# Design OS Validation Report

## Generation Context
- **Design System**: {name from YAML}
- **Page Type**: {landing-page | web-app | dashboard | etc.}
- **Target**: {brief description of what was generated}
- **Generated**: {timestamp}

## Anti-Slop Checklist

### Typography
- [x] No banned font families used outside design system spec
- [x] Heading hierarchy is aggressive
- [ ] ⚠️ FLAG: h2 and h3 sizes too similar (28px vs 24px) — consider increasing h2
- [x] Line-height set intentionally

### Color
- [x] All values reference CSS variables
- [x] No banned gradient patterns
- [ ] ⚠️ FLAG: Section 3 has no colored accent — may feel flat

### Layout
- [x] No three-equal-column card grid
- [x] Hero is not the banned centered pattern
- [x] Sections vary in structure (LP-1, LP-4, LP-5, LP-7, LP-8 used)
- [ ] ⚠️ FLAG: Sections 4 and 5 have identical padding (py-24) — consider varying

### Imagery
- [x] No glassmorphism
- [x] No abstract blobs
- [x] No floating geometry

### Copy
- [ ] ⚠️ FLAG: Feature section 2 headline ("Powerful Analytics") is generic — needs product-specific rewrite
- [x] No placeholder text

### Uniformity
- [x] Border-radius varies between component types
- [ ] ⚠️ FLAG: All cards use the same shadow — consider varying elevation
- [x] Not every section uses the same internal layout

## Visual Verification Needed

The following items require browser-based visual verification that Claude Code cannot perform:

1. **Viewport test**: Load the page at 1440px, 1024px, 768px, and 375px widths. Verify no horizontal overflow, no text truncation, and layout shifts are intentional.
2. **Color contrast**: Verify that text on colored backgrounds meets WCAG AA. Especially check:
   - Primary button text on primary background
   - Muted-foreground text on muted/card backgrounds
   - Any text overlaid on images
3. **Typography rendering**: Verify fonts loaded correctly. Check that heading font and body font are visually distinct (if the design system specifies different families).
4. **Spacing rhythm**: Scroll the full page. Verify the alternating dense/spacious rhythm is visually perceptible.
5. **Interactive states**: Hover over buttons, links, and cards. Verify hover states exist and feel responsive.
6. **Dark mode** (if applicable): Toggle dark mode. Verify all sections remain legible and no raw colors leak through.

## Flagged Issues Summary

| # | Severity | File | Line | Issue |
|---|----------|------|------|-------|
| 1 | low | src/components/sections/features.tsx | 12 | h2/h3 size gap too small |
| 2 | low | src/app/page.tsx | 45 | Sections 4-5 identical padding |
| 3 | medium | src/components/sections/features.tsx | 28 | Generic headline needs rewrite |
| 4 | low | src/components/sections/pricing.tsx | 8 | All cards same shadow |

Severity levels:
- **high**: Anti-slop rule violation. Must fix before shipping.
- **medium**: Quality concern. Should fix.
- **low**: Suggestion for improvement. Optional.
```

## Cursor Validation Workflow

When Cursor receives the project with the validation report:

### Step 1: Read the Report
Open `VALIDATION-REPORT.md` and review all flagged issues.

### Step 2: Browser Visual Check
1. Start the dev server (`npm run dev`).
2. Navigate to the generated page.
3. Take a screenshot at desktop width (1440px).
4. Take a screenshot at mobile width (375px).
5. Compare against the design system identity:
   - Does the page feel like the specified design system (Halo = purple/rounded, Ember = orange/monospace headings, Nitro = blue/sharp)?
   - Does the typography hierarchy feel deliberate and aggressive?
   - Does the layout vary between sections?
   - Does the color usage feel intentional and sparse?

### Step 3: Fix Flagged Issues
Address issues in the report by severity (high first, then medium).

### Step 4: Slop Gut Check
The final test is subjective: does this page look like it was designed by a human with opinions, or does it look like it was generated by an AI that averaged 10,000 SaaS templates? If the latter, identify what specific elements create that feeling and fix them.

Common tells to watch for even after anti-slop enforcement:
- Every section has the same visual weight.
- The page feels "safe" — nothing is bold, nothing is unusual, nothing would make a designer have an opinion.
- Spacing is too uniform even if individual values technically vary.
- The page could belong to any product in any industry.

### Step 5: Sign Off
If the page passes visual verification, remove or archive `VALIDATION-REPORT.md`. The page is ready.
